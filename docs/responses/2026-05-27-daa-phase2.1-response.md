## Condition 1: Define Validation Algorithm

**DAA Concern:** "Intent hash" and "entity extraction" undefined, latency budget unknown

**Response:**
- **Intent hash:** SHA-256 of normalized query
  - Normalization: lowercase, strip whitespace, sort entities
  - Example: `"What is 2+2?"` → `"what 2 2 is"` → hash
- **Entity extraction:** Regex-based (v1.6), LLM-based (v1.7)
  - Patterns: URLs, code snippets, numbers, dates
  - Overhead: 5-10ms
- **Validation decision tree:**
  1. Check intent hash match (fast path, ~5ms)
  2. If mismatch, extract entities and compare (~15ms)
  3. If still mismatch, validation fails → fresh inference
- **Latency budget:** <50ms total (p95)
  - Embedding: 30ms
  - Entity extraction: 10ms
  - Comparison: 10ms

**Implementation:** `layercache/cache/validator.py`

## Condition 2: Specify Probation Parameters

**DAA Concern:** N value, comparison algorithm, performance impact unquantified

**Response:**
- **N = 10 requests** (statistical significance without excessive cost)
- **Comparison algorithm:** Embedding cosine similarity (threshold 0.95)
  - Reuses existing FastEmbed infrastructure
  - Overhead: ~30ms per probation check
- **Probation storage:** `probation_count` column in `semantic_cache` table
- **Memory bounds:** Max 1000 entries in probation (LRU eviction)
- **Auto-promotion:** After 1 hour, entries auto-promote regardless of count
- **Cost:** $0.06 per new cache entry
  - 10 requests × 2K tokens × $0.003/1K tokens
- **Asynchronous probation:** Serve cache immediately, validate in background
  - Trade-off: faster (no latency impact), risk of serving bad responses
  - Mitigation: retroactive invalidation if probation fails

**Implementation:** `layercache/cache/probation.py`

## Condition 3: Document Cache Coherency Model

**DAA Concern:** Hierarchy, invalidation protocol, race conditions

**Response:**
- **Hierarchy:** Cascading (must hit semantic first, then verify provider)
- **Lookup order:**
  1. Semantic cache (embedding similarity ≥0.95)
  2. If hit, check `provider_cache_valid` flag
  3. If flag=false, skip provider cache, go to inference
  4. If flag=true, proceed with provider cache markers
- **Invalidation protocol:**
  - Template update → invalidate semantic cache entries for that template
  - Provider cache → auto-expires per provider TTL (Anthropic: 5min-1hr, OpenAI: 5-10min)
  - No manual provider cache invalidation needed
- **Race condition handling:**
  - Concurrent requests for same entry: first wins (SQLite row locking)
  - Probation tracking: atomic increment (`probation_count = probation_count + 1`)

**Implementation:** `layercache/cache/tier.py`

## Condition 5: Define Fallback Behavior

**DAA Concern:** What happens when validation fails?

**Response:**
- **Validation failure path:**
  1. Log validation failure (INFO level, includes `intent_hash`, `validation_stage`)
  2. Mark cache entry as `probation_failed` (don't increment `probation_count`)
  3. Fall back to fresh inference
  4. Store fresh response in cache (new entry, reset probation)
- **User impact:** ~50ms latency overhead (validation time), then normal inference
- **Metrics:** Track `validation_failures_total` counter for monitoring

**Implementation:** `layercache/cache/validator.py:validate()`

## Condition 6: Quantify Probation Cost

**DAA Concern:** Dollar cost per new cache entry

**Response:**
- **Cost formula:** `N × avg_tokens × cost_per_token`
- **Assumptions:**
  - N = 10 requests
  - avg_tokens = 2,000 (input + output)
  - cost_per_token = $0.003/1K (OpenAI GPT-4o)
- **Calculation:** 10 × 2,000 × $0.003/1,000 = **$0.06 per new cache entry**
- **Anthropic cheaper:** 10 × 2,000 × $0.0025/1,000 = $0.05 per entry
- **Break-even:** Entry needs ~200 uses to offset probation cost
  - At 200 uses: $0.06 probation / 200 = $0.0003 per request
  - Without cache: $0.006 per request (2K tokens × $0.003/1K)
  - Savings: 95% after break-even

**Dashboard metric:** `probation_cost_per_entry` (rolling average)

## Condition 7: Add Feature Flag

**DAA Concern:** Safe rollout, ability to disable

**Response:**
- **Config option:** `caching.multi_tier.enabled` (default: true)
- **Feature flag behavior:**
  - `true`: Use multi-tier cache (semantic → prefix → inference)
  - `false`: Use single-tier semantic cache only (v1.5 behavior)
- **Hot-reloadable:** Config change takes effect immediately (no restart)
- **Dashboard toggle:** Admin UI can flip flag (with confirmation)
- **Metrics:** Track `multi_tier_enabled` gauge for monitoring

**Implementation:** `layercache/config.py`, `layercache/pipeline.py`

**Current:** Already exists in `models.py:232`

**Clarification:**
- `lc_skip_semantic_cache`: Skip lookup, but store result (warm cache)

- Dashboard metrics: separate counters (`bypass_total`, `skip_semantic_total`)

**Problem:** Cold start for new templates/users

**Solution (v1.7):**

- Proactive embedding generation for common queries
- Out of scope for v1.6 (additive improvement)
