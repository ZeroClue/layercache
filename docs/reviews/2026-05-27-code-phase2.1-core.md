## Performance

**Latency Analysis:**

- ✅ Intent hash validation: <10ms average (test: `test_validation_fast_path`)
- ✅ Entity extraction: 5-10ms estimated (regex-based, no LLM)
- ✅ p95 latency: <50ms (test: `test_validation_latency_budget` — 100 iterations)
- ✅ Probation increment: O(1) with SQLite index
- ✅ LRU eviction: O(log n) with index on `created_at`

**Risk Assessment:**

- ⚠️ `evict_oldest()` called on every `increment_probation_count()` adds latency spike when max_entries reached
- ⚠️ No connection pooling for `ProbationTracker._db` (single connection, may bottleneck under load)
- ✅ WAL mode enabled for concurrent reads
- ✅ Indexes created for all query patterns

**Memory:**
- No in-memory caching (all SQLite-based)
- Bounded by `max_probation_entries` (default 1000)
- No memory leaks detected

## Blocking Issues

None. All critical paths implemented and tested.

## Non-Blocking Suggestions

1. **Optimize eviction:** Move `evict_oldest()` from `increment_probation_count()` to a background task or call only every N increments

2. **Add connection pooling:** Use `aiosqlite.Pool` or similar for high-concurrency scenarios

3. **Improve entity extraction:** Current implementation replaces query with entities only. Should preserve original normalized query and append entities for better hash stability

4. **Add metrics:** Track probation promotions, evictions, validation failures in `MetricsCollector`

5. **Document trade-offs:** Add comment explaining why embedding comparison was simplified to intent hash for v1.6

6. **Add health check:** Expose `ProbationTracker.healthy` status in dashboard metrics endpoint

7. **Consider Redis backend:** For high-scale deployments, probation tracking may need Redis instead of SQLite
