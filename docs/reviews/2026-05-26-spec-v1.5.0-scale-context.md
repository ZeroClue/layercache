## Spec Review: v1.5.0 Scale & Context Intelligence

### Verdict
⚠️ Approve with conditions

### Conditions

1. **R1 — Session ID in prefix_hash()**: The spec states session isolation is required, but `StratifiedPrompt.prefix_hash()` must explicitly include `session_id` in the hash calculation. Verify this is implemented in `layercache/models.py`.

2. **R1 — Health check timeout**: Spec mentions "2s timeout" for Redis ping but `redis_timeout` config is 5s. Clarify if these are separate timeouts (connection vs. health check).

3. **R2 — Token counter fallback**: Spec says `tiktoken` is default but doesn't specify behavior when tiktoken fails or model is unknown. Add fallback strategy.

4. **R3 — Redis key collision**: Analytics and cache use same Redis DB with different prefixes. Document what happens if prefix collision occurs (e.g., `layercache:cache:metrics:*`).

5. **P1 — Gemini persistence**: Spec notes this is a "bug fix" but includes it in P1 scope. Either move to separate bugfix release or explicitly state it's a P1 deliverable with its own tests.

### Strengths

- **Clear problem statement**: Four specific gaps identified (SQLite scaling, session isolation, truncation, visibility)
- **Testable requirements**: Each requirement has explicit acceptance criteria with checkboxes
- **Provider research included**: Caching mechanism analysis prevents duplicating provider effort
- **Phased implementation**: Realistic 4-phase plan with clear file lists
- **DA conditions tracked**: Spec already includes conditions table showing responsiveness to feedback
- **Out of scope explicit**: v1.6+ items clearly deferred

### Concerns

| Section | Issue | Severity |
|---------|-------|----------|
| R1 — Redis Backend | No mention of Redis AUTH/password support in config | Medium |
| R1 — Fallback | SQLite fallback on Redis failure could cause data inconsistency if partial writes occurred | Medium |
| R2 — Truncation | `important` strategy scoring formula is heuristic; no validation that it actually preserves "important" content | Low |
| R2 — Token counting | `tiktoken` adds dependency; not listed in "Files to Modify" | Low |
| R3 — Analytics | No specification for how "per-agent" tracking works if session IDs are auto-generated | Medium |
| R4 — Gemini TTL sync | "TTL sync" is mentioned but implementation details unclear (which TTL wins if mismatch?) | Medium |
| P1 Timeline | 2 days for Redis + session isolation + Gemini fix seems optimistic | Low |
| Tests | "20+ new tests" is vague; should specify test categories (unit, integration, load) | Low |

### Recommendations

1. **Add Redis auth config**: Extend `SemanticCacheConfig` with `redis_password_env` field for production deployments.

2. **Clarify fallback semantics**: Document that SQLite fallback is for *read* operations only during Redis outage, or implement write-ahead logging.

3. **Specify agent tracking**: If session IDs auto-generate, explain how dashboard will show "per-agent" metrics (client must send consistent session ID).

4. **Add tiktoken to requirements**: Explicit dependency listing prevents import errors.

5. **Define TTL sync behavior**: State that Redis TTL should match Gemini `CachedContent` TTL exactly, with Redis TTL being the source of truth.

6. **Split Gemini fix**: Consider cherry-picking Gemini persistence fix to v1.4.x if it's truly a bug fix (data loss on restart).

### Missing Edge Cases

- **Redis connection pool exhaustion**: What happens when all 10 connections are in use? Spec says "block 5s, then fallback" but doesn't specify if pending requests queue or fail.

- **Session ID header injection**: If client sends malicious `X-Session-ID` value (e.g., very long, special chars), is it sanitized before use in Redis key?

- **Truncation during active conversation**: If budget is 50K tokens and conversation is at 49K, adding one message triggers truncation. Does this cause cache miss for what was previously cached?

- **Concurrent cache invalidation**: Two agents invalidating same prefix hash simultaneously — is this handled atomically?

- **Analytics rollup failure**: If background rollup computation fails, is there retry logic or data loss?

### Security Notes

| Issue | Recommendation |
|-------|----------------|
| Session ID in Redis key | Sanitize session ID before use (alphanumeric + dash only) |
| Redis URL in config | Support env var substitution for `redis_url` to avoid hardcoding passwords |
| Analytics data exposure | Dashboard should require authentication if exposed to network |
| Truncation strategy config | Validate truncation strategy enum at config load time |

### Feasibility Assessment

**Overall: Feasible**

- Redis backend: Standard pattern, `redis.asyncio` is mature
- Session isolation: Simple hash key extension
- Smart truncation: `recent` strategy is trivial; `important` requires testing
- Analytics: Aggregation queries are straightforward; dashboard is incremental

**Timeline: Realistic with caveats**

- P1 (2 days): Tight but doable if Gemini fix is minimal
- P2 (1 day): Reasonable for `recent` + `important` strategies
- P3 (1.5 days): Depends on dashboard complexity
- P4 (0.5 days): Documentation often takes longer than expected
