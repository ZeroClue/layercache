## Executive Summary

LayerCache v1.5.0 adds Redis backend, session isolation, smart truncation, and cache analytics for multi-agent deployments. Scope reduced from initial OAuth epic to focus on core value: prompt/context engineering + token caching.

**Effort:** ~4-5 days  
**Priority:** High (blocks multi-agent concurrent usage)  
**Risk:** Low (incremental over v1.4.0)  

---

## Problem Statement

LayerCache v1.4.0 works for single-agent setups but has three gaps for multi-agent, high-volume usage:

1. **SQLite doesn't scale** — File locking blocks concurrent writes. Multiple agents (Claude Code + Cline + Cursor) contending for the same cache will see degraded performance.

2. **No session isolation** — Cache is shared across all requests with same prefix. Agent A's conversation history can pollute Agent B's cache if L0-L2 match.

3. **Dumb truncation** — Current session truncation just cuts L2 to fit `max_session_tokens`. Loses recent/important context, keeps old/stale content.

4. **No visibility** — Can't see which templates, enhancements, or agents actually save tokens. Flying blind on optimization.

---

## Provider Caching Research

We analyzed how major providers handle caching to avoid duplicating effort:

| Provider | Mechanism | Min Tokens | TTL | Scope | Cost |
|----------|-----------|------------|-----|-------|------|
| **Anthropic** | `cache_control` markers (explicit or auto) | 1K-4K (model-dependent) | 5min (default), 1hr (2x cost) | Per API key + workspace | Write: 1.25-2x base, Read: 0.1x base |
| **OpenAI** | Automatic prefix routing | 1K tokens | 5-10min inactive (max 1hr), 24hr extended (gpt-5.x) | Per org | No extra cost |
| **Gemini** | `CachedContent` explicit resources | N/A | Configurable (default 1hr) | Per API key | Standard pricing |

**Key findings:**

1. **All providers cache by API key** — No session isolation at provider level
2. **Exact prefix match required** — Any change invalidates cache
3. **Lookback window limits** — Anthropic: 20 blocks, OpenAI: hash-based routing
4. **Cache is NOT shared across orgs/workspaces** — Provider-level isolation exists

**Implication for LayerCache:**
- Provider cache handles **exact prefix matches** within a session
- LayerCache should handle **semantic matches** across sessions
- Two-layer strategy: LayerCache (semantic, session-aware) → Provider (exact, API-key-shared)

## Requirements

### R1 — Redis Backend with Session Isolation

**Acceptance Criteria:**
- [ ] Two agents can hit cache concurrently without locking
- [ ] Cache entries isolated by session ID
- [ ] Fallback to SQLite if Redis unavailable (dev mode)
- [ ] All existing TTL logic preserved
- [ ] Redis connection pooling configured (max 10 connections, 5s timeout)
- [ ] Health check before fallback (ping Redis with 2s timeout)

**Config:**
```yaml
caching:
  semantic:
    backend: redis  # or "sqlite"
    redis_url: redis://localhost:6379/0
    redis_pool_size: 10  # Max connections
    redis_timeout: 5  # Connection timeout (seconds)
    db_path: /tmp/layercache.db  # fallback
    session_isolation: true  # Default: isolate by session
    session_id_header: "X-Session-ID"  # Client sends this
    session_id_auto_generate: true  # Auto-generate if missing (stored in cookie/header)
```

**Cache Key Structure:**
```
redis_key = "layercache:cache:{session_id}:{prefix_hash}"
```

**Implementation:**
- `layercache/cache/redis.py` — Redis backend implementing same interface as `SemanticCache`
- Factory function: `get_cache_backend(config)` returns Redis or SQLite instance
- Pipeline extracts session ID from header, cookie, or generates UUID (stored in response header for reuse)
- Redis health check: ping with 2s timeout before falling back to SQLite
- Connection pool: `redis.asyncio.ConnectionPool(max_connections=10, socket_timeout=5)`

**SQLite Fallback Limitations (to be documented):**
- File locking blocks concurrent writes
- Not production-ready for multi-agent deployments
- Suitable for: dev mode, single-agent testing, demo environments

### R2 — Smart Truncation

**Acceptance Criteria:**
- [ ] Truncation keeps recent context, drops old gracefully
- [ ] Configurable strategy via YAML
- [ ] Budget enforcement with graceful degradation (no hard failures)
- [ ] Token counting uses tiktoken (provider-agnostic)

**Config:**
```yaml
caching:
  max_session_tokens: 50000  # Token budget for L2
  truncation_strategy: recent  # or "important" | "semantic" (v1.6)
  token_counter: tiktoken  # Default: tiktoken (provider-agnostic)
```

**Strategies:**

| Strategy | Behavior | Use Case | v1.5? |
|----------|----------|----------|-------|
| `recent` (default) | Keep last N messages that fit budget | General conversation | ✅ |
| `important` | Score by: length (chars/4) + tool_calls×2 + system_keywords×3 | Tool-heavy workflows | ✅ |
| `semantic` | Embed messages, drop least similar to current query | RAG-like scenarios | ❌ v1.6 |

**`important` Strategy Scoring Formula:**
```python
score(message) = len(message.content) / 4  # Approximate tokens
                 + (2 if message.has_tool_calls else 0)
                 + (3 if any(kw in message.content for kw in ['system', 'instruction', 'context']) else 0)
```

**Truncation Flow:**
```
1. Request received → stratify into L0-L4
2. Count L2 tokens (tiktoken)
3. If over budget → truncate L2 BEFORE cache lookup
4. Compute prefix_hash() on TRUNCATED L0-L2
5. Lookup cache with truncated prefix
6. On miss → call LLM → store with truncated prefix
```

**Note:** Truncation changes L2, which changes `prefix_hash()`. This is intentional — truncated prompts have their own cache namespace.

**Implementation:**
- `layercache/truncation.py` — `TruncationStrategy` enum + implementations
- `Truncator` class with `truncate(prompt, budget, strategy)` method
- Pipeline calls truncator BEFORE cache lookup if L2 over budget
- Token counter: `tiktoken.get_encoding('cl100k_base')` (works across providers)

### R3 — Cache Analytics

**Acceptance Criteria:**
- [ ] Dashboard shows hit rates per agent/model/template
- [ ] Per-enhancement ROI (does CoT actually save tokens?)
- [ ] Prometheus metrics exported
- [ ] Query performance <100ms for dashboard views
- [ ] Redis key prefixes separated (cache vs metrics)

**Metrics:**
- LayerCache hit rate (%) — semantic cache hits / total requests
- Provider cache hit rate (%) — requests with `cache_read_input_tokens > 0`
- Token savings — estimated tokens saved via LayerCache vs direct provider call

**Redis Key Structure for Analytics:**
```
layercache:metrics:raw:{timestamp}     # Raw events (TTL: 7 days)
layercache:metrics:hourly:{date}:{hour}  # Hourly rollups (TTL: 30 days)
layercache:metrics:daily:{date}       # Daily rollups (TTL: 365 days)
```

**Dashboard View:**
- Time series: hit rate over time (hourly/daily)
- Top templates by savings
- Agent comparison (if session IDs tracked per agent)
- Model breakdown (which models benefit most from caching)

**Implementation:**
- `layercache/metrics/aggregator.py` — Aggregation queries (hourly/daily rollups)
- `layercache/dashboard/templates/analytics.html` — Analytics UI
- Pre-compute rollups in background snapshot loop (not real-time)
- Redis key prefixes: `layercache:cache:*` vs `layercache:metrics:*`

### R4 — Provider Cache Alignment

**Acceptance Criteria:**
- [ ] Gemini `CachedContent` map persisted to Redis (fix in-memory loss on restart)
- [ ] LayerCache doesn't duplicate provider cache unnecessarily
- [ ] Clear documentation on when to use LayerCache vs provider cache
- [ ] Gemini TTL sync: Redis TTL matches `CachedContent` TTL

**Design:**
- **LayerCache**: Semantic match on L4 (user query), session-isolated
- **Provider**: Exact prefix match on L0-L2, shared by API key
- **Complementary**: LayerCache avoids API call entirely; provider reduces token cost within session

**Gemini Fix:**
- Current: `_cache_map` is in-memory dict, lost on restart
- Fix: Store `prefix_hash → cached_content_name` mapping in Redis
- TTL: Match `CachedContent` TTL (default 1hr)
- Redis key: `layercache:gemini:{prefix_hash}` with TTL sync

**Note:** This is a bug fix (data loss on restart), not a feature. Should be tested separately and potentially cherry-picked to v1.4.x if users report issues.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Redis adds deployment complexity | Medium | Medium | SQLite fallback, Docker Compose includes Redis |
| Smart truncation breaks cache hits | Low | High | Make strategy configurable, default to `recent` |
| Analytics queries slow | Low | Low | Pre-compute rollups, cache results |
| Session ID generation adds latency | Low | Low | UUID4 is fast (~100ns), negligible vs LLM call |
| Truncation invalidates cache | Medium | Medium | Documented behavior — truncated prompts have own cache namespace |
| SQLite concurrent write failures | High (if misused) | High | Document limitations clearly; health check before fallback |
| Redis pool exhaustion | Low | Medium | Block 5s, then fallback; configurable pool size |

## Implementation Notes (DA Concerns Addressed)

**Truncation Timing:**
- Truncation happens BEFORE cache lookup
- Truncated L2 → different `prefix_hash()` → separate cache namespace
- Flow: request → stratify → count L2 tokens → truncate if over → hash → lookup

**Session ID Persistence:**
- Auto-generate UUID once per client (not per-request)
- Store in response header: `X-Session-ID: <uuid>`
- Client should reuse this header for subsequent requests
- Alternative: store in cookie (for browser-based clients)

**Redis Connection Pooling:**
- Pool size: 10 connections (configurable via `redis_pool_size`)
- Timeout: 5s socket timeout, 2s ping timeout for health check
- On pool exhaustion: block for 5s, then fallback to SQLite

**Analytics Retention:**
- Raw events: 7 days (TTL on Redis keys)
- Hourly rollups: 30 days
- Daily rollups: 365 days

**Token Counter:**
- `tiktoken.get_encoding('cl100k_base')` — works across OpenAI, Anthropic, Gemini
- Approximate: 4 chars ≈ 1 token (used for `important` strategy scoring)

**`important` Strategy Scoring:**
```python
score = len(content) / 4  # Approximate tokens
        + (2 if has_tool_calls else 0)
        + (3 if any(kw in content for kw in ['system', 'instruction', 'context']) else 0)
```
